# AWS Bedrock AgentCore Provider

This lab registers AWS Bedrock AgentCore as an AgentRegistry **Runtime**. It is a two-step pattern: (1) create an IAM role in your AWS account from a CloudFormation template generated by `arctl`, and (2) register that role as a Runtime in AgentRegistry.

## Lab Objectives

- Generate the CloudFormation template with `arctl provider setup aws`
- Deploy the stack and capture `RoleArn` + `ExternalId`
- Apply the AWS Runtime manifest
- Verify with `arctl get runtimes`

## Prerequisites

- [040 â€” `arctl` authenticated](040-arctl-auth.md)
- AWS account with `iam:CreateRole` and `cloudformation:CreateStack` permissions
- `aws` CLI installed and configured

## 1. Generate the CloudFormation Template

```bash
arctl provider setup aws --aws-account-id <YOUR_AWS_ACCOUNT_ID> > /tmp/agentregistry-cf.yaml
```

The template creates an IAM role with permissions for:

| Service | Why |
|---------|-----|
| Bedrock AgentCore | Create/manage agent runtimes |
| IAM | Create execution roles for agents |
| S3 | Upload agent code artifacts |
| CloudWatch Logs | Agent logging |
| AppConfig | Agentgateway configuration |
| Cognito | (Optional) agent auth |
| EC2 | (Optional) managed agentgateway instances |

The template prints the **External ID** and **Role Name** at the bottom â€” note them; you will need them in Step 3.

## 2. Deploy the Stack

```bash
aws cloudformation create-stack \
  --stack-name agentregistry-access-role \
  --template-body file:///tmp/agentregistry-cf.yaml \
  --capabilities CAPABILITY_NAMED_IAM \
  --region us-east-1

aws cloudformation wait stack-create-complete \
  --stack-name agentregistry-access-role \
  --region us-east-1

aws cloudformation describe-stacks \
  --stack-name agentregistry-access-role \
  --region us-east-1 \
  --query 'Stacks[0].Outputs'
```

Capture the `RoleArn` and `ExternalId` from the output.

## 3. Register the AWS Runtime

```bash
export AWS_ROLE_ARN="<RoleArn from CloudFormation output>"
export AWS_EXTERNAL_ID="<ExternalId from CloudFormation output>"
export AWS_REGION="us-east-1"

cat > /tmp/aws-runtime.yaml <<EOF
apiVersion: ar.dev/v1alpha1
kind: Runtime
metadata:
  name: AWS
spec:
  type: BedrockAgentCore
  config:
    roleArn: "${AWS_ROLE_ARN}"
    externalId: "${AWS_EXTERNAL_ID}"
    region: "${AWS_REGION}"
EOF

arctl apply -f /tmp/aws-runtime.yaml
arctl get runtimes
```

> **For tracing,** also set `spec.telemetryEndpoint` to the **external** address of the AgentRegistry OTel Collector (AWS AgentCore runs outside your cluster, so it can't reach `*.svc.cluster.local`). See [090](090-observability-tracing.md) for that variant.

You should see a Runtime named `AWS` with `type: BedrockAgentCore`.

## 4. Updating AWS Credentials

If you used long-lived IAM user keys in your Helm values, rotating them is a Helm upgrade:

```bash
# Edit /tmp/are-values.yaml with new accessKeyId / secretAccessKey
helm upgrade agentregistry-enterprise \
  oci://us-docker.pkg.dev/solo-public/agentregistry-enterprise/helm/agentregistry-enterprise \
  --version 2026.5.3 \
  --namespace agentregistry-system \
  -f /tmp/are-values.yaml \
  --wait --timeout 5m
```

The server pod rolls automatically when the AWS Secret changes.

## Troubleshooting

### CloudFormation stack fails

```bash
aws cloudformation describe-stack-events \
  --stack-name agentregistry-access-role \
  --region us-east-1 \
  --query 'StackEvents[?ResourceStatus==`CREATE_FAILED`]'
```

Common causes:

- IAM user lacks `iam:CreateRole` or `cloudformation:CreateStack`
- Role name collision â€” pass `--role-name <unique>` to `arctl provider setup aws`

### `arctl provider setup aws` requires authentication

Yes â€” it calls the AgentRegistry API. Log in first ([040](040-arctl-auth.md)) or pass `--registry-token <token>`.
